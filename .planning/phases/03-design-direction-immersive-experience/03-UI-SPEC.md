# Phase 3 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

Scope: WOW-02 (career scroll-storytelling), WOW-03 (micro-interactions + seamless transitions, one engine + motion tokens), WOW-04 (skippability, no preloader), MODE-02 (reduced-motion quiet variant), TECH-02 (deliberate mobile experience). Source of locked decisions: `03-CONTEXT.md` (D-01…D-19). This contract resolves the "Claude's Discretion" items from CONTEXT with concrete, prescriptive values so planning/execution do not re-ask.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — no shadcn, no component-library initialization this phase |
| Preset | not applicable |
| Component library | none — hand-authored Tailwind 4 + GSAP components (standing project convention, confirmed: no `components.json`, no headless-UI dependency in `package.json`) |
| Icon library | none — hand-authored inline SVG, `currentColor` fills/strokes (matches existing header-cluster convention noted in `layout.tsx`) |
| Font — Body/Label | Geist Sans + Geist Mono, both via `next/font/google` (existing, self-hosted at build — zero runtime Google call, satisfies AGENTS.md) |
| Font — Display (NEW, D-03) | **Bricolage Grotesque** (variable, weight/width/optical-size axes), via `next/font/google` — free, open-source, in the Google Fonts catalog, self-hosted at build like Geist. Chosen within the locked "technical grotesque" category: designed specifically to gain character at large display sizes (tightens/sharpens above ~48px) while staying legible — exactly the "characterful, pairs with Geist, engineered without going full terminal" brief. Less ubiquitous in dev portfolios than Space Grotesk, avoiding a templated feel. Reference: fonts.google.com/specimen/Bricolage+Grotesque. |

**Font usage discipline:** Geist Sans stays the body/paragraph face. Geist Mono stays the label/metadata/annotation face (nav, eyebrows, dates, tags, spine ticks). Bricolage Grotesque is used **only** for the Display role (hero name, case-study H1, and any large standalone headline moment) — never for body copy (it is explicitly not optimized under ~24px per its own design brief).

## Spacing Scale

Declared values (must be multiples of 4) — unchanged from the shipped 4px/8pt Tailwind default scale already in use (`gap-*`, `px-*`, `py-*` throughout `page.tsx`):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, chip inline padding (`px-2 py-0.5`) |
| sm | 8px | Compact element spacing, tag/chip gaps |
| md | 16px | Default element spacing, list-item gaps |
| lg | 24px | Section-internal padding, grid gutter |
| xl | 32px | Layout gaps between sub-blocks |
| 2xl | 48px | Sub-section breaks |
| 3xl | 64px+ | Page-level section breaks (existing `gap-20 sm:gap-28` = 80/112px, an intentional exception, see below) |

Exceptions:
- Hero/section vertical rhythm already ships at 80px/112px (`gap-20`/`sm:gap-28`) — larger than the 3xl token but still a multiple of 4/8; kept as-is, not a violation.
- Structural grid lines / spine rail / chip borders use **1px hairline** widths — border-width tokens are exempt from the spacing scale (they are line weights, not spacing).
- Icon-only interactive controls (e.g. a collapsed mobile Contact affordance) get a **44×44px minimum touch target** regardless of the 8pt rhythm of their visible glyph, per existing WCAG-driven convention in `layout.tsx`. Any such icon-only control MUST carry an accessible name — the collapsed mobile Contact affordance uses `aria-label="Kontakt" / "Contact"` (from the existing `nav.contact` content key, locale-aware), with the SVG glyph marked `aria-hidden="true"`. Execution must not ship an unlabeled icon button.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px (`text-base`, browser default) | 400 (Regular, Geist Sans) | 1.6 |
| Label (mono: nav, eyebrows, dates, tags, spine ticks) | 12px (`text-xs`) | 500 (Medium, Geist Mono) | 1.4 — eyebrows add `uppercase tracking-[0.25em]` |
| Heading (h3 role/project/skill sub-headings) | 18px (`text-lg`) | 500 (Medium, Geist Sans) | 1.3 |
| Display (h1 hero name, case-study H1, standalone headline beats) | Fluid `clamp(2.75rem, 2rem + 5vw, 6rem)` (44px → 96px) | 700–800 (Bold/ExtraBold via Bricolage's variable weight axis, tuned tighter width/optical-size at the top of the clamp) | 1.05 |

**Weight discipline (body/UI content):** exactly 2 operative weights on Geist — Regular 400 (paragraphs, descriptions) and Medium 500 (labels, role titles, skill names, sub-headings). `font-semibold` (600) is **retired from the hero H1** in this phase (that role migrates to the Display face/weight above) and is not reintroduced elsewhere unless a proven gap arises. Display is its own role with its own weight scale (Bricolage variable axis), which does not count against the 2-weight UI budget. **For an auditor's tally:** total distinct weight values in use across the site is **3** — 400 and 500 on the Geist UI faces (the enforced 2-weight UI budget), plus the single Display role rendered on Bricolage's variable weight axis (700–800 treated as one role, not two discrete weights). This is a disciplined role-scoped exception, not unconstrained weight noise.

**Section H2 convention (unchanged, load-bearing):** section headers (`career`, `projects`, `skills`, `about`, `activity`, `contact`) keep the existing **mono-eyebrow treatment** (`text-xs uppercase tracking-[0.25em] text-muted`, i.e. the Label role) rather than becoming large Display headings. This is intentional — it reinforces the engineered/console motif (section headers read like console labels, not magazine titles) and must not be "upgraded" to Display size during Phase 3 execution.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--background)` (`#fafaf9` light / `#0a0a0a` dark) | Page background, all surfaces |
| Secondary (30%) | `var(--muted)` (secondary text) + `var(--border)` (structural rules) | De-emphasized text (descriptions, dates, meta); hairline grid lines, spine rail, dividers, tech-chip outlines. **No filled card/surface color exists or is introduced** — the engineered look is built from background + rules, not colored panels. |
| Accent (10%) | `var(--accent)` (`#c2410c` light / `#fb923c` dark) | See reserved list below — signal color only |
| Destructive | not applicable this phase | No destructive actions exist in this phase (no forms, no delete flows — REQUIREMENTS Out of Scope) |

**Accent reserved for (explicit, exhaustive):**
1. CV download button fill (existing, unchanged)
2. `:focus-visible` ring (existing, unchanged — must not regress TECH-03)
3. Active/current progress-spine tick + its fill segment (D-07)
4. External/case-study link text on hover-adjacent inline links (existing `text-accent` pattern on org links, case-study "→" links)
5. Magnetic-interaction affordance (a subtle accent-tinted glow/underline at max pull, never the resting state)
6. One flourish character per split-text headline reveal is permitted to briefly accent-flash during the reveal stagger (optional, disciplined — never the whole headline)

No other element may use the accent color. This list is exhaustive, not illustrative.

## Motion Tokens & Choreography (D-08 — single source of truth, GSAP + `@gsap/react`)

Central CSS custom properties (added to `globals.css` alongside the existing color tokens), read by GSAP via `getComputedStyle` or mirrored as a TS constants module (`src/lib/motion-tokens.ts`) consumed by `useGSAP` hooks — pick one source of truth in planning, do not duplicate values in two places.

| Token | Value | Usage |
|-------|-------|-------|
| `--motion-duration-fast` | 150ms | Hover/focus/active micro-transitions |
| `--motion-duration-base` | 400ms | Standard reveals (fade/slide-in-on-enter) |
| `--motion-duration-slow` | 800ms | Hero intro beats, section-to-section transitions |
| `--motion-duration-chapter` | 1200ms | ITSC multi-beat sub-sequence chapter transition (D-06) |
| `--motion-ease-out` | `cubic-bezier(0.16, 1, 0.3, 1)` | Default entrance/reveal easing (expo-out) — matches the `--ease-out-expo` convention already used project-wide |
| `--motion-ease-in-out` | `cubic-bezier(0.65, 0, 0.35, 1)` | Section/route transitions |
| `--motion-ease-magnetic` | `cubic-bezier(0.33, 1, 0.68, 1)` | Magnetic pull + snap-back |
| `--motion-distance-sm` | 8px | Hover micro-lift |
| `--motion-distance-md` | 24px | Reveal-on-enter slide offset |
| `--motion-distance-lg` | 48px | Chapter-level offset (ITSC sub-scene beats) |
| `--motion-stagger-char` | 18ms | Split-text char-by-char stagger |
| `--motion-stagger-word` | 40ms | Split-text word-by-word stagger |
| `--motion-stagger-list` | 60ms | List/card reveal stagger (career roles, tech chips, bento cells) |

All motion animates `transform`/`opacity` only (compositor-friendly, per project performance rules) — no `width`/`height`/`top`/`left`/`margin`/`padding` animation.

**Lenis ↔ ScrollTrigger sync (D-09, standard integration recipe):**
```
lenis.on('scroll', ScrollTrigger.update);
gsap.ticker.add((time) => lenis.raf(time * 1000));
gsap.ticker.lagSmoothing(0);
```
Mount once in a client boundary near the root of `[locale]/layout.tsx` (or a dedicated `<MotionProvider>` wrapping `children`), guarded so it never runs during SSR.

**Reduced-motion gate (D-09, D-18 — hard requirement):** `prefers-reduced-motion: reduce` disables Lenis entirely (no `lenis.raf` calls, native scroll) and disables all GSAP timelines (see Reduced-Motion Contract below). Checked once via `matchMedia` at mount; the codebase already has this exact pattern for the no-flash theme script — follow the same "safe, static, no external call" discipline.

**Touch/mobile gate (D-19 — resolved discretion item):** Lenis is **active on `pointer: fine` only** (desktop/trackpad). On `pointer: coarse` (touch), Lenis is **not instantiated at all** — native touch scroll is used, which already feels good and avoids the battery/INP cost of lerping touch input on mid-tier Android. GSAP ScrollTrigger reveals are unaffected by this (they observe scroll position; they do not require Lenis to function), so the reveal timeline still works identically on touch. Magnetic effects (D-11.1) use the same `pointer: fine` gate and are simply absent (not degraded) on touch/keyboard.

**Seamless transitions technique (D-11.4 — resolved discretion item):** **GSAP-orchestrated crossfade**, not the View Transitions API. Rationale: (a) keeps the single-engine discipline (D-08) — introducing a second transition system for one feature contradicts the phase's own success criterion; (b) React's experimental View Transition primitives are not yet a stable, production-safe fit for this all-static Next.js 16 site — revisit only if the GSAP approach proves genuinely insufficient. Pattern: on `Link` click to a case study or `/about`, a client transition wrapper (`useGSAP`) fades/slides the outgoing content out (`--motion-duration-base`, `--motion-ease-in-out`) before navigation commits, and the incoming page runs its own standard mount-reveal (same reveal-on-enter primitives as scroll sections) — a coordinated exit + timed entrance, not true shared-element morphing.

## Component Specs

### Hero intro sequence (D-12)
Static SSR HTML (name/role/value-prop/nav) is already the LCP element and is visible with zero JS. On mount (client-only enhancement):
1. `0–400ms`: grid/tick overlay lines draw in (`scaleX`/`scaleY` reveal, `--motion-ease-out`)
2. `200–900ms`: H1 name (now Display face) reveals via GSAP SplitText, per-word stagger (`--motion-stagger-word`)
3. `500–1000ms`: mono value-prop line fades + rises (`--motion-distance-md`, `--motion-duration-base`, `--motion-ease-out`)

Nav links and any interactive element remain focusable/clickable throughout — no `pointer-events: none` gate during the sequence (WOW-04: nothing may block the fast path, even mid-animation). Reduced motion: sequence collapses to `0ms`, all elements render at final state on mount (D-18).

### Progress spine (D-07)
- Visible `lg:` (1024px) and up only. Below `lg`, chapter/year context is conveyed inline per career entry (dates already render there) — no information loss, matches D-19's "focused single column, not shrunk desktop."
- Structure: fixed vertical 1px line (`var(--border)`) in the left margin column, scoped to the Career section's scroll range only (not the whole page).
- Ticks: one per career org entry, small horizontal dash + mono label formatted **`{2-digit chapter index}·{year}`** (e.g. `01·2019`, `02·2021`). Decorative marks are `aria-hidden="true"`; the information is already present in the visible DOM dates, so no duplicate screen-reader text is needed.
- Active tick: label color `var(--foreground)`, dash color `var(--accent)`; inactive ticks: label + dash `var(--muted)`.
- Progress fill: an accent-colored segment grows from the top of the rail to the current scroll position within the Career section (GSAP ScrollTrigger `onUpdate`, transform-based, not width/height).

### Bento project grid (D-14)
- Mobile (`<640px`): single column. ELIA and Vidama render full-width with a heavier top border (`2px var(--accent)` — a controlled second use is NOT introduced; use `2px var(--border)` plus a slightly larger Heading size to signal "featured" without spending extra accent budget) and their full summary; compact projects stack tightly below.
- Tablet (`640–1024px`): ELIA full-width row, Vidama full-width row beneath it, compact projects in a 2-column grid below.
- Desktop (`≥1024px`, 12-col grid): ELIA spans `8/12` with a supporting tag/metric panel filling the remaining `4/12` (asymmetric bento cell); Vidama mirrors the pattern, offset to alternate visual rhythm (D-04 "asymmetric placement"); compact projects render in a dense `4/12`-per-item row beneath.
- Reveal: bento cells stagger in on scroll-enter using `--motion-stagger-list`.

### Designed hover/focus/active states (D-11.3)
| Element | Rest | Hover | Focus | Active/press |
|---------|------|-------|-------|---------------|
| Nav/footer links | `text-muted` | `text-foreground`, `--motion-duration-fast` | existing `:focus-visible` ring (2px accent, 2px offset) — unchanged | n/a |
| Org/project title links | `text-foreground` | `text-accent`, `--motion-duration-fast` | ring, unchanged | n/a |
| Tech/skill chips | `border-border` | `border-foreground` + subtle `background: color-mix(in oklab, var(--foreground) 5%, transparent)` | ring on focus if interactive | n/a |
| CV download button | `bg-accent` | `bg-foreground` (existing) | ring, `outline-offset-4` (existing) | `scale(0.98)`, `--motion-duration-fast` |
| Magnetic buttons/links (D-11.1) | static | pulls toward cursor within a 60px radius, max 12px displacement, `--motion-ease-magnetic` | unaffected — ring still applies on keyboard focus even though magnetic pull is pointer-only | snaps back on release/leave |

### Photo treatment (D-16 — design now, asset later)
- Square (1:1) frame, `160px` (mobile) → `200px` (desktop).
- `1px solid var(--border)` frame with 4 small L-shaped accent-colored corner tick marks (inline SVG or CSS `::before`/`::after`), echoing the coordinate/grid motif.
- Image filter: `grayscale(20%) contrast(105%)` — a restrained duotone-adjacent treatment, not full desaturation.
- Mono caption slot beneath the frame (short identity tag, e.g. role/location) — copy TBD by content owner, not blocking.
- Layout: left-aligned beside About text on `lg:` and up (2-column within the reading column); stacked above text on mobile. Section degrades cleanly to text-only if no image is supplied (existing Phase 2 behavior, unchanged).

### Reserved 3D seam (D-13 — resolved discretion item: slot component, not a raw canvas)
A named, empty client-boundary component, e.g. `<HeroSceneSlot />`, rendered as an `absolute inset-0 -z-10 pointer-events-none` layer behind the hero content (`aria-hidden="true"`, no visual output in Phase 3). Phase 4 fills its children with a lazily-mounted `<Canvas>` (react-three-fiber) — no hero re-layout, no z-index/positioning changes needed at that point.

## Reduced-Motion Contract (MODE-02, D-18 — same DOM, motion stripped)

| Motion feature | Reduced-motion equivalent |
|---|---|
| Lenis smooth scroll | Disabled entirely — native scroll |
| GSAP ScrollTrigger reveals | Elements render at final opacity/position immediately (no animation, no delay) |
| Hero intro sequence | Collapses to 0ms — final state on mount |
| Split-text headline reveals | Text renders instantly, no per-char/word stagger |
| Magnetic buttons | Disabled (already pointer-only; also gated off here) |
| Progress spine | Renders statically (current position still shown, no animated fill transition) |
| Section/route transitions | Instant swap, no crossfade |

This is **one implementation**, not a separate "calm mode" — same DOM, same content, same layout; only the animation layer is stripped, checked once via `matchMedia('(prefers-reduced-motion: reduce)')` at mount (no user-facing toggle, per D-17).
